# TODO: Signal 模板单文本 DSL 方案

> 关联文档：`doc/todo/TODO_param_dependency_enum.md`
>
> 这两篇文档理论上属于同一个任务的两个子议题：
>
> * Signal 模板 DSL 升级
> * 参数依赖 DSL 升级
>
> 两者不应割裂设计，原因是：
>
> * Signal 文本 DSL 需要数值表达式 parser
> * 参数依赖 DSL 也需要数值表达式 parser
> * 更合理的方案是共享底层 numeric expression parser，而不是各写一套
>
> 因此，阅读和执行时应将两篇文档视为同一任务下的关联设计。

本文档描述一套新的 Signal 模板语法设计方案。目标是把当前基于 `SignalTemplate + SignalGroup + comparisons/sub_groups` 的树状配置，收敛为一个**单文本 DSL**。

这套 DSL 的核心目标很明确：

*   一个策略只写一段纯文本
*   支持变量定义
*   支持数值计算
*   支持递归逻辑组合
*   最终显式赋值四个信号出口

但这里必须先把边界写死：

*   **这不是一个通用脚本语言**
*   **这不是一个开放式模板系统**
*   **它必须是强约束的最小 DSL**
*   **它必须追求唯一写法，而不是表达力最大化**

也就是说，未来的 Signal 模板不再强调“Python 里手写很多 `SignalGroup` 对象”，而是强调：

*   写一段结构清晰的模板脚本
*   交给 Rust 解析
*   由 Rust 顺序执行变量求值和信号生成

它的定位只能是：

*   **服务项目当前最优先、最常见的策略模板场景**

而不能是：

*   希望靠一套 DSL 覆盖未来所有可能的表达需求

否则最终结果一定是：

*   语法越来越松
*   分支规则越来越多
*   validator 越来越复杂
*   维护成本越来越高
*   模板语义越来越脏

这条原则必须作为最高约束保留下来。

---

## 1. 为什么要改

当前信号模板的核心结构是：

*   `SignalTemplate`
*   `SignalGroup`
*   `comparisons: Vec<String>`
*   `sub_groups: Vec<SignalGroup>`

这种结构的问题不是不能用，而是越来越不适合复杂策略：

*   Python 侧模板很啰嗦
*   中间变量无法表达
*   数值表达式无法表达
*   逻辑组合依赖手写嵌套 `sub_groups`
*   越复杂的策略越像在拼 AST，而不是写模板

你现在提出的方向，本质上是把 Signal 模板从“树状配置”升级成“顺序脚本 DSL”。这个方向是对的。

---

## 2. 总体设计结论

新的 Signal 模板推荐直接改成：

*   **一个纯文本脚本**
*   **分号作为语句终止符**
*   **支持 `#` 注释**
*   **支持变量赋值**
*   **支持数值表达式与布尔表达式**
*   **最后显式给四个出口变量赋值**

也就是说，模板作者写的是：

1. 先定义数值变量
2. 再定义布尔变量
3. 再做递归逻辑组合
4. 最后给四个信号出口赋值

这个流程天然比现在的 `SignalGroup` 递归树更顺。

但这里必须强调：即使改成单文本，也不是要把它做成“可随意编程”的东西。

正确方向是：

*   **只保留最小必要能力**
*   **只保留最稳定的写法**
*   **只保留项目当前高频真实需求**

如果某种能力不是高频刚需，默认就不支持，而不是“先做进去再说”。

---

## 2.1 设计哲学：严格约束、唯一写法、只服务优先场景

这一节必须单独写清楚。

### 2.1.1 严格约束

DSL 不应该追求“想怎么写都行”，而应该追求：

*   语法面尽量小
*   歧义尽量少
*   一眼可审阅
*   一旦写错立即报错

也就是说：

*   不靠宽松容错提升体验
*   不靠多种等价写法提升灵活性
*   而是靠唯一正确写法提升稳定性

### 2.1.2 唯一写法

同一种语义，应尽量只允许一种推荐写法。

例如：

*   语句必须用 `;`
*   最终出口必须显式赋值
*   逻辑组合必须使用 `!`、`&`、`|`
*   不鼓励再保留另一套平行模板风格

唯一写法的价值不是“强迫症式统一”，而是：

*   减少文档复杂度
*   减少 parser 分支
*   减少 code review 成本
*   减少策略模板在不同人手里越写越散

### 2.1.3 只服务优先场景

这套 DSL 的定位应当非常克制：

*   只服务项目当前最常见的研究策略模板
*   只服务高频比较、过滤、触发、组合场景
*   只服务能稳定映射到当前回测引擎语义的表达

不应为了“可能未来会用到”而提前支持：

*   过深的元编程能力
*   任意脚本控制流
*   自定义函数系统
*   任意副作用语义

原因非常直接：

*   这种 DSL 一旦走向通用化，就会极难维护
*   语义一旦松动，bug 会非常难定位
*   解析、校验、执行三层都会迅速膨胀

一句话：

*   **宁可能力少，也不要做成难维护、容易出问题的半成品脚本语言。**

## 3. 推荐的最终形态

推荐把当前的树状模板升级成单文本 DSL，例如：

```text
# 数值变量
sma_pct_5m = sma_0, ohlcv_5m, 0 / close, ohlcv_5m, 0;
rsi_mid_ok = ($rsi_lower < rsi, ohlcv_5m, 0) & (rsi, ohlcv_5m, 0 < $rsi_upper);
trend_ok = macd, ohlcv_1h, 0 > macd_signal, ohlcv_1h, 0;

# 最终信号
entry_long = (sma_pct_5m > 1.01) & rsi_mid_ok & trend_ok;
exit_long = sma_pct_5m < 0.99;
entry_short = false;
exit_short = false;
```

这个形式的几个优点非常明显：

*   人能顺着读
*   中间量可以命名
*   表达式可以复用
*   最终出口一眼就能看到

## 4. 语句规则

### 4.1 语句终止

每条语句必须以 `;` 结尾。

建议规则写死：

*   **分号是唯一语句终止符**
*   **换行只负责排版，不负责语义**

这样做的好处是：

*   多行排版自由
*   解析器更稳定
*   不依赖缩进或换行风格

### 4.1.1 无歧义设计原则

这套 text DSL 应补一个明确的语法原则：

*   **空格和换行不参与语义**
*   **只要保留 token、括号和分号，whitespace normalize 后语义必须保持一致**

也就是说，以下两段应当语义完全等价：

```text
a = close, ohlcv_5m, 0 / sma_0, ohlcv_5m, 0;
entry_long = (a > 1.01) & trend_ok;
```

```text
a=close,ohlcv_5m,0/sma_0,ohlcv_5m,0;
entry_long=(a>1.01)&trend_ok;
```

这个原则非常适合本项目的“强约束、唯一写法、最小 DSL”方向，因为它能带来：

*   更稳定的 parser
*   更容易做 normalize / formatter
*   更容易做快照测试和文本等价比较
*   更少的排版风格歧义

但要做到这一点，DSL 必须主动避免会和标识符粘连的歧义 token。

### 4.2 注释

使用 `#` 表示单行注释。

规则：

*   `#` 后到该行结尾的内容忽略
*   纯注释行不需要分号

### 4.3 语句类型

初版只需要支持两类语句：

1. 变量赋值
2. 最终信号赋值

形式统一都是：

```text
name = expr;
```

差别只在于：

*   普通变量名是用户自定义标识符
*   最终信号名是固定保留字

## 5. 最终信号出口

建议最终仍然沿用当前业务命名，而不是重新发明一套名字：

*   `entry_long`
*   `exit_long`
*   `entry_short`
*   `exit_short`

原因很简单：

*   和现有 `SignalTemplate` 字段一一对应
*   减少命名迁移成本
*   不制造 `long_enter / entry_long` 两套表述

为了保持明确性，建议要求：

*   **四个出口必须显式赋值**

如果某一边不用，就直接写：

```text
entry_short = false;
exit_short = false;
```

不要依赖“缺失即 false”的隐式规则。

---

## 6. 表达式类型

DSL 里应当存在两类表达式：

1.  数值表达式
2.  布尔表达式

### 6.1 数值表达式

数值表达式用于：

*   中间数值变量
*   比较左右侧
*   派生数值计算

支持内容：

*   数值字面量：`1`、`0.5`
*   参数引用：`$rsi_lower`
*   数据操作数：`close, ohlcv_5m, 0`
*   变量引用：`sma_pct_5m`
*   四则运算：`+ - * /`
*   括号：`(...)`
*   常用函数：`min()`、`max()`、`abs()`、`floor()`、`ceil()`、`round()`

这里必须明确一条硬约束：

*   **基础数值运算必须支持递归嵌套**
*   **必须支持用 `()` 显式区分优先级**

也就是说，不只是支持：

*   `a + b * c`

还必须支持：

*   `(a + b) * c`
*   `(a + (b * c)) / d`
*   `floor((x + y) / 2)`
*   `(close, ohlcv_5m, 0 / sma_0, ohlcv_5m, 0) - 1`

原因很简单：

*   这是量化交易策略中的常见场景
*   比例、归一化、均值、价差、偏离度等表达都会频繁使用
*   如果不支持递归嵌套，只会迫使模板拆出大量无意义中间变量
*   最终会让 DSL 既别扭又难维护

这里还要补一条：

*   **函数调用可以保留**
*   **但必须强制使用 `name(...)` 形式**

原因是函数名虽然也是英文标识符，但只要 grammar 明确规定：

*   标识符后紧跟 `(` 才解释为函数调用
*   不支持隐式乘法
*   不支持表达式无运算符直接并列

那么函数调用本身就是无歧义的。

### 6.2 布尔表达式

布尔表达式用于：

*   中间布尔变量
*   最终信号出口

支持内容：

*   比较表达式：`a > b`
*   区间表达式：`in_zone(a, b, c)`
*   进入区间表达式：`enter_zone(a, b, c)`
*   交叉表达式：`cross_up(a, b)`、`cross_down(a, b)`
*   逻辑非：`!expr`
*   逻辑与：`expr & expr`
*   逻辑或：`expr | expr`
*   括号递归组合：`(a & b) | (!c & d)`
*   布尔字面量：`true`、`false`
*   布尔变量引用

这就足够支持你说的：

*   `(&)|(&)` 任意递归逻辑组合

这里必须更新一个关键设计结论：

*   **字母型中缀运算符不应继续保留**

例如：

*   `in`
*   `xin`
*   `x>`
*   `x<`
*   `x>=`
*   `x<=`
*   `x==`
*   `x!=`

这些写法在去掉空格后会天然和标识符粘连，破坏“空白不参与语义”的设计原则。

因此，更合理的方案是：

*   基础比较保留为符号运算符：`> < >= <= == !=`
*   特殊业务语义统一改成函数式写法

例如：

*   `cross_up(a, b)`
*   `cross_down(a, b)`
*   `cross_ge(a, b)`
*   `cross_le(a, b)`
*   `in_zone(a, low, high)`
*   `enter_zone(a, low, high)`

## 7. 数据操作数：保留现有结构，但不再支持省略写法

新的 DSL 不需要推翻现有 data operand 的基本结构，但应当收紧写法约束。

统一只允许这种完整写法：

*   `close, ohlcv_5m, 0`
*   `rsi, ohlcv_1h, 1`
*   `sma_0, ohlcv_15m, &1-3`

也就是说，在新 DSL 中应当明确：

*   **不再支持省略 source**
*   **不再支持省略 offset**
*   **所有数据操作数都必须显式写完整来源信息**

因此，以下旧式短写法都不应继续支持：

*   `close`
*   `close, ohlcv_5m`
*   `rsi`

这样收紧的原因是：

*   变量系统已经存在，重复书写问题可以通过变量解决
*   一旦允许省略来源，AST 中的来源信息就需要靠默认值推断
*   来源推断越多，后面交叉校验、映射校验、调试诊断就越容易变脏
*   强制显式来源后，模板语义会更稳定，也更容易审阅

一句话：

*   **既然已经支持变量来简化模板，就不应该再靠省略来源来换短写法。**

所以这次更新的重点不是改 data operand 结构，而是：

*   保留现有 data operand 结构
*   取消省略写法
*   在其外面包一层“脚本语言壳”
*   允许它进入数值表达式
*   允许中间变量复用

## 8. 变量系统

### 8.1 变量类型

变量允许保存两类结果：

*   数值结果
*   布尔结果

从使用角度看，用户不需要手工声明类型，系统可以在 Rust 里做语义推导。

### 8.2 变量引用规则

变量只能引用已经在前面定义过的变量。

也就是说：

*   **采用顺序求值**
*   **禁止前向引用**

这和前面参数依赖的思路一致，也符合人写模板时的直觉：

*   先定义中间量
*   再用中间量

### 8.3 变量命名规则

变量名建议限制为普通标识符：

*   `sma_pct_5m`
*   `trend_ok`
*   `rsi_mid_ok`

不建议允许：

*   带空格
*   带 `.` 的变量名
*   和保留字冲突

---

## 9. 运算符优先级

为了避免歧义，优先级建议明确写死：

### 9.1 数值表达式优先级

*   函数调用 / 括号
*   一元负号
*   `* /`
*   `+ -`

并且这里的“括号优先”必须建立在递归表达式语法之上，而不是只做浅层包裹。

也就是说，numeric expression parser 应支持真正的递归嵌套树，而不是线性 token 拼接。

### 9.2 布尔表达式优先级

*   比较 / 区间 / 交叉
*   `!`
*   `&`
*   `|`

也就是说：

```text
a > b & c > d | e > f
```

应被解释为：

```text
((a > b) & (c > d)) | (e > f)
```

但即使如此，仍然建议在复杂逻辑里强制写括号，保持模板可读性。

---

## 10. 类型与广播规则

这套 DSL 不只是 parser 问题，还涉及运行时类型规则。

建议明确以下规则：

### 10.1 数值广播

如果比较两侧是：

*   一个 Series
*   一个 Scalar

则自动做 scalar broadcast。

这和当前 signal evaluator 的行为保持一致。

### 10.2 布尔运算

`&` 和 `|` 只允许作用于布尔表达式。

不允许：

*   数值直接参与逻辑与或

### 10.3 最终信号出口

`entry_long / exit_long / entry_short / exit_short` 必须解析为布尔序列。

如果右侧不是布尔表达式，直接报错。

## 11. 交叉与区间语义在新 DSL 下的约束

引入变量和算术后，交叉类运算符会比现在更敏感，因此约束必须保留，而且要更严格。

### 11.1 交叉语义

当前规则是：

*   交叉类语义只能作用于 base 时间轴

这个规则在新 DSL 下仍然成立。

也就是说，即使你写的是变量：

```text
fast_gap = close, ohlcv_5m, 0 - sma_0, ohlcv_5m, 0;
entry_long = cross_up(fast_gap, 0);
```

只要这个变量底层来自 base 时间轴，就允许。

但如果变量底层来自高周期映射，仍然不允许把它拿去做交叉。

### 11.2 需要引入“来源标记”

这意味着数值表达式在 Rust 内部不只是一个值，还必须携带一份简化的来源元数据，例如：

*   是否来自 base 时间轴
*   是否包含高周期映射
*   是否包含区间 offset

由于新 DSL 会强制每个 data operand 显式写出 `name, source, offset`，这些来源信息就不应再靠默认补全后猜测，而应直接从解析后的 AST 明确获得。

这样在做：

*   `cross_up(...)`
*   `cross_down(...)`
*   `enter_zone(...)`

等操作时，才能继续做和今天一样的语义校验。

## 12. 应该和参数表达式 DSL 如何复用

这一点非常关键。

既然参数依赖也要引入字符串表达式 DSL，那么最合理的做法不是写两套完全独立的数值表达式 parser，而是：

*   抽一层共享的 **numeric expression parser**

然后：

*   参数 DSL 直接复用 numeric expression parser
*   Signal DSL 在 numeric expression parser 之上，再加 comparison / logic / statement

也就是说，推荐分成三层：

1.  **共享层**
    * 数字字面量
    * 标识符
    * 参数引用
    * 四则运算
    * 函数调用
    * 括号

2.  **参数 DSL**
    * 只关心 numeric expression

3.  **Signal DSL**
    * numeric expression
    * comparison
    * bool logic
    * statement
    * final signal slots

这才是真正合理的“复用”。

## 13. 解析与执行顺序

推荐执行流程如下：

1.  先去掉注释
2.  再删除所有空白字符（空格、Tab、换行）
3.  按分号切分语句
4.  逐条 parse 成 statement AST
5.  顺序执行 statement
6.  把变量值写入环境
7.  最后收集四个出口变量

也就是说，这个 DSL 的执行模型应当是：

*   **顺序脚本**
*   **单次遍历**
*   **无副作用重赋值**

初版建议直接禁止变量重复赋值，避免语义复杂化。

### 13.1 先清理，再解析

这套 DSL 的解析流程应明确采用：

*   **先清理文本**
*   **再解析语句**

推荐预处理顺序：

1.  删除 `#` 注释
2.  删除所有空白字符
3.  按 `;` 切 statement
4.  对每个 statement 单独 parse

这样做的好处是：

*   parser 更简单
*   “空白不参与语义”的原则可以真正落地
*   statement 是天然的最小语义单元
*   后续报错和测试都更轻

### 13.2 报错定位：使用语句序号，不使用行号

这套 DSL 不建议维护行号级定位。

原因：

*   语义分隔符是 `;`
*   换行只是排版
*   预处理会删除空白和换行
*   行号在这个 DSL 中不稳定，也没有太高价值

因此，推荐的错误返回信息应当是：

*   `statement_index`
*   `raw_statement`
*   `reason`

也就是说，报错时只需要带：

*   **语句序号**
*   **原始语句文本**

这已经是最轻量、最简单、也最实用的定位方式。

例如：

```text
Signal DSL 解析失败
statement_index: 7
raw_statement: entry_long = cross_up(fast_gap, 0) & trend_ok;
reason: unexpected token ...
```

这比维护复杂的行列号映射更符合当前 DSL 的设计哲学。

---

## 14. Breaking Change 建议

这套方案如果上了，建议直接做破坏性更新，不要保留旧的 `SignalGroup` 递归配置作为长期兼容层。

原因：

*   两套模板系统长期并存，代码一定会变脏
*   parser、validator、resolver、文档都会分叉
*   后续策略模板会出现两种风格混用

更合理的方式是：

*   新 DSL 设计稳定后
*   直接统一迁移
*   不保留长期双轨

这是更符合项目代码品味的做法。

## 15. PyO3 轻量入口

除了 DSL 本身，这里还应补一个配套工程约束：

*   **通过 PyO3 导出更轻量的 signal 引擎入口**

原因很现实：

*   当前很多 signal 相关测试必须沿整条 backtest 路径进入
*   测试准备过重
*   失败定位过深
*   想单测 parser / validator / resolver / evaluator 时不够直接

因此，建议补一个更轻量的 PyO3 入口，用于直接驱动 signal 系统，而不是每次都先构造完整 backtest。

### 15.1 直接收益：方便测试

当前最直接的价值就是：

*   更容易写 signal parser 单测
*   更容易写 signal evaluator 单测
*   更容易写变量、逻辑组合、交叉约束的契约测试
*   报错定位更直接

也就是说，短期第一价值不是“开放公共 API”，而是：

*   **让测试从重链路变成轻链路**

### 15.2 但入口本身应保持一定通用性

虽然当前动机主要是方便测试，但这个入口不应该被设计成“只能给测试凑合用”的临时接口。

更合理的目标是：

*   **优先服务轻量测试**
*   **同时保持一定通用性**

因为 signal 解析与求值系统本身是有一定独立价值的：

*   它不完全等同于 backtest
*   它本身就是一层模板解析与布尔信号生成能力
*   未来理论上可能被其他场景复用

例如未来可能出现：

*   只想检查某个模板在给定数据上的信号结果
*   只想做 parser / validator 工具
*   只想输出信号布尔列，而不跑完整回测

因此这个 PyO3 入口应该：

*   先以测试友好为第一目标
*   但接口命名和职责不要做成一次性测试 hack

### 15.3 推荐入口形态

建议考虑导出一类轻量 API，例如：

*   `parse_signal_text(...)`
*   `validate_signal_text(...)`
*   `run_signal_engine(...)`

其中最有价值的是最后一个：

*   输入：数据、指标结果、参数、signal text
*   输出：中间变量结果、四个最终信号列、必要的诊断信息

这样可以让测试直接落在 signal 层，而不是被迫穿过完整回测链。

### 15.4 边界要求

即使开放 PyO3 signal 入口，也必须保持边界清晰：

*   signal 入口只负责 signal parser / validator / resolver / evaluator
*   不要偷偷再塞进 backtest 语义
*   不要为了“通用”把接口做得过大

也就是说：

*   **它可以是独立的**
*   **但仍然要保持最小职责**

---

## 16. 最终结论

本任务的推荐方案如下：

*   Signal 模板升级为 **单文本 DSL**
*   使用 `;` 作为唯一语句终止符
*   使用 `#` 作为注释
*   支持变量赋值
*   支持数值表达式
*   支持比较、区间、交叉
*   支持 `!`、`&`、`|` 任意递归逻辑组合
*   强制显式赋值 `entry_long / exit_long / entry_short / exit_short`
*   新 Signal DSL 与参数 DSL 共享底层 numeric expression parser
*   这套 DSL 必须长期坚持“强约束、最小能力、唯一写法、只服务优先场景”
*   同时通过 PyO3 导出轻量 signal 入口，优先解决测试过重问题，并保留适度通用性

一句话概括：

*   **Signal 模板从“树状配置”升级为“强约束的顺序脚本模板”。**

这会比当前写很多 `SignalGroup` 和 `comparisons` 更自然，也更适合你现在想要的模板库方向。
