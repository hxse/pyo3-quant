# 重复时间戳与 Renko 清理

## 1. 当前正式答案已经是什么

`03-10` 之后，这个问题其实已经没有悬念。

当前正式答案就是：

1. `time` 列必须存在。
2. `time` 列必须是 `Int64`。
3. `time` 列不允许为空。
4. `time` 列必须严格递增。
5. 不支持重复时间戳 source。

所以 `04-05` 不是要“重新判断是否支持 Renko”，而是把旧残余清干净。

## 2. 当前残余分布

1. 生产代码残余：`py_entry/data_generator/*` 仍保留 Renko 入口。
2. Rust 残余：stitched 检查仍保留“非递减即可”的旧兜底。
3. 文档与测试残余：结构文档、测试注释、辅助帮助函数和测试参数构造仍把 `renko_*` 当可接入口径。

## 3. 本任务的正式选择

1. 删除 Renko 入口，不保留“能生成但下游拒绝”的过渡状态。
2. stitched 与结构文档统一收口到严格递增。
3. 测试只保留一般性的“重复时间戳 source 必须 fail-fast” contract。
4. 测试注释、示例和帮助文本也要同步去掉 `renko_*` 的正式示例地位。

这里的“收口到严格递增”不是只改文档，Rust 侧也要一起收紧：

1. 删除 stitched 中旧的 `assert_source_times_non_decreasing(...)` 语义。
2. 把非 base source 也统一纳入严格递增校验。
3. 失败信息应明确到具体 `source_key` 与问题位置，避免留下“文档很严、实现还在放水”的双轨。

## 4. 为什么要把注释和测试辅助层也算进来

因为对当前仓库来说，旧痕迹不只是在生产代码里。

如果测试辅助函数和注释还在把 `renko_*` 当正式示例，它们一样会误导后续实现与审阅。
