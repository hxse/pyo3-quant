# AGENTS.md - 通用协作守则与技术指南

本文件定义一套可跨项目复用的协作协议。核心目标是反对氛围编程，采用严格的 `spec-driven` 工作流，以清晰的真值链、明确的阶段 Gate、可审阅的报告体系来约束实现质量。

## 1. 沟通与语言规范
*   **全程中文沟通**：所有分析、讨论、Plan、Review 报告、Spec 报告均必须使用中文。
*   **代码与注释语言约束**：代码本身（变量名、函数名、类名、模块名等）必须使用英文；代码注释必须使用中文。
*   **强制注释**：写代码时必须带上必要的中文注释，优先解释控制流、关键约束、边界条件与容易误读的实现点。
*   **报告风格**：审阅报告默认按固定顺序输出：`文档清单 -> 总体判断 -> Findings`；若无问题，必须明确写出“未发现阻断问题”，并补充残余风险、验证缺口或未覆盖原因。
*   **审阅归属判断**：进行审阅时，除指出问题本身外，还必须补充说明该问题属于“只改代码”“只改文档”还是“代码与文档一起改”。
*   **审阅报告沉淀边界**：代码审阅报告与文档审阅报告默认只停留在讨论对话中，不本地沉淀；本地只沉淀已经进入代码执行、验收或修复结果的内容。

## 2. 核心研发范式
*   **开发顺序**：
    *   先统一场景、目标、范围、术语、架构、阶段步骤、控制流程、关键算法、边界条件、输入输出契约与失败语义，再进入实现。
    *   反对氛围编程，反对“先写再说、边写边想”。
*   **术语统一**：
    *   仓内统一使用 `spec`。
    *   `spec` 表示任务快照中的正式解释层与真值载体，不把它当成长期维护的全局百科。
*   **方案优先**：
    *   每次改动先确认方案，再进入代码落地。
    *   高耦合、不可视、容易 quietly wrong 的链路，必须先有足够清晰的 Spec，再允许实现。
*   **禁止生造范式**：
    *   进入实现前必须先查找并复用仓库已有范式、入口、抽象、注册机制、配置方式与测试方式。
    *   禁止为了当前任务局部方便，平行发明第二套架构、第二套计算链路、第二套验证方式、第二套类型系统、第二套错误系统或第二套配置系统。
    *   若现有范式暂不支持需求，必须先明确报告“现有范式缺口”，并在 Spec 或方案中说明如何扩展既有范式；不得静默绕开既有链路。
    *   找不到已有能力时，默认先停下来核对上下文或提出最小假设；禁止把临时实现伪装成项目正式范式。
*   **真值优先级**：
    *   单次任务的正式真值载体是该任务的最新 Spec 快照。
    *   若不同任务快照冲突，以最新任务快照与当前源代码为准。
    *   若任务快照与当前源代码冲突，以当前源代码为准，但必须在审阅报告里明确指出不一致。
*   **默认收口方式**：
    *   默认目标是收口到唯一正式设计，不长期保留旧新并存的双轨状态。
    *   默认倾向破坏性更新，Spec 面向最新设计状态，不为旧兼容服务。
    *   新设计确定后，默认不保留旧接口、旧桥接、旧别名、旧术语或临时兼容写法。
    *   若必须保留兼容层，必须在 `02_spec` 与 `03_execution` 中写清理由、范围、退出条件与清理时机。

## 3. Task Snapshot 原则
*   **任务快照而非全局百科**：
    *   不维护一套持续回改、持续追踪“是否过时”的全局大 Spec。
    *   每次任务只维护该任务自己的 Spec 快照与执行记录。
    *   历史任务默认不回改，不承担“始终全局最新”的责任。
*   **Task Spec 默认存放位置**：
    *   Task Spec 快照默认放在 `doc/tasks/`。
    *   每个任务使用单独文件夹承载。
*   **轻量全局索引**：
    *   建议维护单一索引文件，如 `doc/tasks/TASK_INDEX.md`。
    *   索引只记录“任务名 + 5~10 句摘要”。
    *   索引用于检索上下文，不承担版本治理与真值裁决职责。
*   **任务命名规范**：
    *   `task_name` 必须包含日期，推荐格式：`YYYY-MM-DD <title>`。

## 4. Task 分级制度
*   **除特别小的改动外，每个任务必须先定级，再开始写 Spec 或改代码**。
*   **AI 在进入一个新任务时，必须先判断并明确声明当前任务级别**。
*   **若任务未定级，禁止进入执行阶段**。
*   **任务级别必须写入该任务的 `00_meta.md` 文件**，作为该任务的正式上下文入口。
*   **`00_meta.md` 必须收束为单一文件**：
    *   不再使用 `00_meta/` 目录，也不再拆分多个 meta 文件。
    *   这是任务元信息的唯一正式入口，其他部分仍使用文件夹组织。
*   **每个正式 Task 都必须明确写出“范围内（In Scope）”与“范围外（Out of Scope）”**。
    *   不能只写笼统目标，必须明确这次任务负责解决什么、不解决什么。
    *   “非目标”若表述较抽象，仍需进一步落到清晰的范围外清单。
    *   若执行中发现新增需求落在原范围外，必须先更新 Task Spec 或拆成新 Task，再继续执行。

### 4.0 极小改动豁免
*   **特别小的改动可以不走 Task 流程，允许直接改动**。
*   **特别小的改动默认不记录在 Task 中**。
*   **适用范围必须同时满足以下条件**：
    *   改动范围局部且清晰。
    *   不涉及架构调整、流程变化、关键算法、公开契约、接口风格、类型系统、状态传播或 breaking change。
    *   不需要专门的方案讨论与阶段化执行。
    *   可以直接从当前源码与上下文一眼确认真值。
*   **若执行过程中发现复杂度上升、影响范围扩大，或出现关键不明确点，必须立即升级为正式 Task，并补齐分级与 Spec 流程**。

### 4.1 A 类任务
*   适用范围：
    *   高耦合链路。
    *   quietly wrong 风险高。
    *   牵涉核心真值链、关键数据流或状态流、跨模块公共契约、共享基础设施、公开接口、类型体系、执行语义或破坏性重构。
*   要求：
    *   完整 Task Snapshot 结构。
    *   必须使用完整文件结构。
    *   完整阶段 Gate。
    *   完整 AI pre-review 与 post-review。
    *   明确分阶段执行与分阶段验收。
    *   明确旧逻辑删除计划与旧痕迹清扫计划。

### 4.2 B 类任务
*   适用范围：
    *   中等复杂度改动。
    *   涉及多个文件，但核心契约较稳定。
    *   需要一定设计说明与执行规划，但不需要超重型论证。
*   要求：
    *   仍然使用任务文件夹。
    *   文件结构允许按复杂度精简。
    *   允许压缩 `01_context` 与 `02_spec` 的篇幅。
    *   仍然必须有阶段计划、AI 审阅与最终验证。

### 4.3 C 类任务
*   适用范围：
    *   局部低风险改动。
    *   可从源码快速确认真值。
    *   不涉及核心契约与关键状态传播。
*   要求：
    *   仍然需要任务快照，但允许极简形态。
    *   文件结构允许进一步精简，不要求完整目录树。
    *   允许使用压缩版 Spec 与执行记录。
    *   若执行过程中发现复杂度上升，必须立即升级到 B 或 A。

## 5. Task Snapshot 文件结构
*   **每个 Task 仍然由单独文件夹承载，而不是散落成多个平级大文件**。
*   **除 `00_meta.md` 外，其余部分默认使用文件夹组织**。
*   **Spec 本身也应拆分，避免形成巨型文件**。
*   **若单个 Spec 文件过大或混入多个职责，应继续拆分**。

### 5.1 推荐文件结构
*   **A 类任务必须使用完整文件结构**。
*   **B 类与 C 类任务可按复杂度精简文件结构**，不要求机械保留所有目录，但 `00_meta.md` 仍应保留。
1. `doc/tasks/<task_name>/00_meta.md`
2. `doc/tasks/<task_name>/01_context/`
3. `doc/tasks/<task_name>/02_spec/`
4. `doc/tasks/<task_name>/03_execution/`
5. `doc/tasks/<task_name>/04_review/`

### 5.2 各部分职责
*   `00_meta.md`
    *   这是任务元信息的唯一正式入口，不再拆成目录与多个文件。
    *   只记录以下三项，不额外扩写其他元信息：
        1. 简单的一段任务描述。
        2. 任务级别定义，以及为什么定这个级别的复杂度说明。
        3. 任务范围，明确什么在任务范围内、什么在任务范围外，并把模糊边界澄清。
*   `01_context/`
    *   记录场景描述、需求分析、问题背景、架构设计思路、不同方案对比、关键取舍、为什么这样设计。
    *   这是“为什么”与“为什么不是别的方案”的主要承载层。
    *   context 阶段必须重点体现设计思路的充分取舍和品味，不能只写背景复述或目标转述。
    *   对关键方案，必须说明候选路径、取舍依据、放弃理由与最终收口原因。
*   `02_spec/`
    *   记录正式设计、术语定义、对象 contract、不变量、边界条件、关键算法、失败语义。
    *   这是任务的正式解释层与真值层。
    *   若某些测试承担正式 contract 证明作用，或属于关键真值链验证，则应同时写入 `02_spec`。
    *   对重构、系统扩展、公共契约调整、测试框架调整等任务，必须写明用于辅助审查的“新增语义逻辑清单”与“冻结语义逻辑清单”。
*   `03_execution/`
    *   记录落地计划、阶段拆分、文件修改清单、模块拆分方案、迁移路径、删除旧逻辑计划、验证计划。
    *   只表达实现需要的内容，不重复 `02_spec` 中已经解释清楚的“为什么”。
    *   一般测试计划、回归测试、执行期验证、命令入口与 diff 核对计划默认写在 `03_execution`。
*   `04_review/`
    *   只记录代码更新对应的结果性信息：执行结果、验收结论、修复记录、残余风险、未决问题。
    *   不记录文档演进过程，不记录 `01_context`、`02_spec`、`03_execution` 的修订轨迹，也不沉淀代码或文档审阅正文。
    *   审阅结论只有在真正进入代码执行、验收或修复结果后，才允许写入 `04_review`。

## 6. `01_context` 与 `02_spec` 的使用规则
*   **`01_context` 写什么**：
    *   写背景、候选方案、比较维度、取舍理由、为什么不用其他方案。
    *   只要存在非平凡设计选择，就必须写清取舍。
    *   若某个决定会影响模块边界、公开契约、失败语义、状态流、迁移路径或旧逻辑删除，必须在 `01_context` 展开。
*   **`02_spec` 写什么**：
    *   只写当前正式设计：contract、不变量、边界条件、关键算法、输入输出、失败语义。
    *   主正文必须按最终态表述，不写设计演化过程。
    *   对未改动区域，只需简要描述当前状态，不必无差别展开。
*   **`02_spec` 禁止什么**：
    *   不混入旧设计、变更轨迹、讨论过程、试探性方案或相对旧版本说明。
    *   不在正式算法、正式契约、对象定义、边界条件、输入输出里使用“改成了……”“原来……现在……”“仍然……”“继续……”“不再……”这类相对旧版本写法。
    *   迁移、废弃、删除旧逻辑，写到 breaking changes 或 `03_execution`，不要反向污染正式 Spec 正文。
*   **`02_spec` 允许什么**：
    *   允许自然语言、步骤化描述、公式、伪代码。
    *   重点是把对象来源、阶段顺序、控制流、失败语义写清楚，不要求贴近可编译语法。
*   **`02_spec` 必须满足什么**：
    *   会影响控制流、报错策略、对象来源、边界分支的语义，必须写清。
    *   不能只写接口骨架。
*   **语义新增与冻结清单**：
    *   对重构、系统扩展、测试框架调整、公共契约调整、执行链路调整等任务，`02_spec` 应声明两类辅助审查清单：
        1. **新增语义逻辑（Added Semantics）**：本任务新增的行为、规则、contract、校验方式、失败语义、输入输出或控制流。
        2. **冻结语义逻辑（Frozen Semantics）**：本任务允许重构实现、拆分文件或重写函数，但语义上必须保持等价的既有行为、入口、输出、失败语义、校验逻辑或控制流。
    *   该清单只是辅助审查的最低审查基线，不是审查边界；pre-review 与 post-review 仍必须进行全面审查，防止代码实现漂移。
    *   若审查时发现清单遗漏了关键语义，必须报告清单不完整，并按影响范围补充 Spec 或拆出新任务。
*   **Task 文件拆分规则**：
    *   一个文件只承担一种主要职责。
    *   除 `00_meta.md` 外，task 下的每个文件都应尽量不超过 `300` 行。
    *   若任一 task 文件同时承担背景、契约、算法、迁移、执行、验收等多种职责，必须拆分。

## 7. 阶段 Gate 与评审机制
*   **Spec Gate**：
    *   `02_spec` 未经人工完整审阅通过，禁止进入执行。
*   **Execution Gate**：
    *   `03_execution` 未完成 AI pre-review，禁止开始代码落地。
*   **Stage Gate**：
    *   A 类任务必须分阶段执行。
    *   每个阶段必须有阶段目标、影响范围、验收条件。
    *   上一阶段未验收通过，禁止进入下一阶段。
*   **Close Gate**：
    *   未完成 post-review、未完成验证、未完成旧痕迹清扫，不得宣称任务完成。

### 7.1 人工审阅与 AI 审阅协作原则
*   **人工审阅负责真值把关**：
    *   尤其是 `01_context` 与 `02_spec` 的逻辑闭环、术语、边界、取舍与 breaking 策略。
*   **AI 审阅负责一致性、遗漏、实现偏差与回归风险排查**。
*   **早期高不确定阶段**：
    *   更依赖人工审阅 Spec，AI 负责辅助修订、结构化与补洞。
*   **后期执行与回归阶段**：
    *   更依赖 AI 自动审阅、自动修复、自动测试与自动回归排查。
*   **无论处于哪个阶段**：
    *   人工仍需审阅 AI 给出的报告。
    *   AI 报告不能代替人工对关键结论的最终确认。
*   **AI 审阅输出顺序**：
    *   固定顺序：`文档清单 -> 总体判断 -> Findings`。
    *   审阅报告默认只在对话中给出，不单独落盘。
*   **文档清单**：
    *   逐个列出本次审阅目标涉及的文档。
    *   每份文档至少写一项：文档状态：`已确认审阅`、`可能遗漏`、`明确未审阅`。
    *   若本轮已经明确做了文档到代码的映射判断，再补一项简短代码状态：`已核对`、`部分核对`、`待核对`。
    *   不要求填写精确代码文件数量，不要求展开对应的代码文件名。
*   **总体判断**：
    *   必须明确回答两句：
        1. 本次审阅目标涉及的文档是否全部已审阅。
        2. 本次审阅目标涉及的代码主链是否已经审阅到位。
    *   若存在 `可能遗漏`、`明确未审阅`、`部分核对` 或 `待核对`，必须说明未覆盖原因。
    *   若无法确认，必须写 `待核对` 或 `可能遗漏`；禁止为了让报告看起来完整而伪造精确数字。
*   **Findings**：
    *   按严重度排序。
    *   每条 `Finding` 固定写三项：
        1. 问题是什么。
        2. 建议怎么改。
        3. 修改归属：`只改代码`、`只改文档`、`代码与文档一起改`。
    *   需要时再补：风险或影响范围、假设或待确认点、以哪里为准。
    *   若没有问题，必须明确写出“未发现阻断问题”，并补充残余风险、验证缺口或未覆盖原因。
*   **这样设计的目的**：
    *   先强迫 AI 检查文档是否漏审、关键代码链路是否漏审，再进入 `Findings`。
    *   允许 `可能遗漏`、`部分核对`、`待核对`，是为了防止 AI 在不确定时伪造完成度。
    *   不强制精确代码数量，是为了避免制造低价值的伪精确字段；格式服务于发现遗漏，不服务于凑数字。

### 7.2 不明确点的报告义务
*   **Spec 阶段**：
    *   若发现关键语义不明确、方案未闭环、边界未冻结、多个方案尚未定夺，必须立即报告。
    *   在这些问题解决前，不得把 Spec 描述成已冻结。
*   **执行阶段**：
    *   若发现关键不明确点会影响控制流、公开契约、删除旧逻辑、模块边界或失败语义，必须显式报告。
    *   只能二选一：
        1. 立即暂停执行并提出问题。
        2. 先基于最小假设继续，但必须在同轮审阅报告里集中列出：假设、风险、待确认点、影响范围。
    *   这里的要求是硬约束，不是建议。
    *   只要用了“最小假设继续”这条路径，就必须把假设显式写出来；不允许自己记在脑子里，不允许等到后面再补，也不允许省略不写。
    *   禁止静默带过。禁止在关键不明确仍然存在时，把实现、Spec 或审阅结论表述成“已经完全明确”或“已经完全通过”。

### 7.3 DoR / DoD
*   **Definition of Ready**：
    *   任务已定级。
    *   `00_meta.md` 已给出简短任务描述。
    *   `00_meta.md` 已明确任务级别，以及为何定为该级别的复杂度说明。
    *   `00_meta.md` 已显式写出范围内与范围外，且两者边界足够清晰，不存在“做到哪算哪”的开放口径。
    *   `01_context` 已充分说明关键方案取舍，而不是只给结论。
    *   `02_spec` 已冻结正式 contract 与边界。
    *   涉及重构、系统扩展、公共契约或测试框架调整时，`02_spec` 已明确用于辅助审查的新增语义逻辑清单与冻结语义逻辑清单。
    *   `03_execution` 已写明阶段计划、文件影响、拆分方案、删除计划与验证计划。
*   **Definition of Done**：
    *   实现与 Spec 一致。
    *   已结合语义清单、`jj diff`（必要时辅以 `git diff`）和全面审查核对实现，未发现未声明的语义漂移。
    *   旧逻辑、旧兼容、旧术语残留已完成清扫。
    *   静态检查通过。
    *   测试通过。
    *   post-review 无阻断问题。
    *   任务目录与索引已回填到位。

## 8. 执行阶段规则
*   **无明确执行命令时，禁止直接改代码**：
    *   先讨论方案、补齐 Spec、明确任务级别与 Gate 状态。
*   **实现前范式核对**：
    *   写代码前必须完成范式核对：确认目标模块已有入口、已有实现模式、已有测试模式、已有错误体系、已有类型来源与已有配置方式。
    *   对核心链路、公共契约、指标计算、回测执行、策略表达、数据处理等高风险区域，未完成范式核对前禁止进入实现。
    *   若核对结果显示需要新增能力，应优先扩展现有正式链路；只有 Task Spec 明确允许时，才可以引入新的架构层或新的执行入口。
*   **执行必须阶段化**：
    *   拆成若干清晰阶段，每阶段聚焦单一目标。
    *   每阶段先立 contract，再扩实现，再做阶段验收。
*   **执行计划必须前置考虑模块化与文件拆分**：
    *   避免先把逻辑堆进大文件，再事后补拆。
    *   必须追求高内聚、低耦合、职责单一。

## 9. 开发工作流
*   **安全删除限制**：
    *   默认避免直接使用不可恢复删除。
    *   若环境支持 `trash` 或其他可恢复删除方式，应优先使用。
    *   若必须做不可恢复删除，需先确认其必要性与影响范围。
*   **仓库入口优先**：
    *   务必先阅读仓库级任务入口，如 `justfile`、`Makefile`、`package.json`、`pyproject.toml`、项目脚本目录等。
    *   优先使用项目已定义的标准命令，而不是临时绕过仓库约定直接调用底层工具。
*   **版本管理入口优先**：
    *   本协作协议默认基于 `jj`（Jujutsu）组织任务变更与本地 patch stack。
    *   若仓库已经启用 `jj`，版本管理、任务拆分、patch stack 整理与 diff 审查都必须优先走 `jj` 工作流；Git 只作为只读辅助或远端兼容层。
    *   若仓库尚未启用 `jj`，允许使用 Git 做状态核对与历史阅读，但默认只允许 Git 只读命令，如 `git status`、`git diff`、`git log`、`git show`、`git blame`。
    *   未启用 `jj` 时，除非用户明确授权，AI 不得执行 Git 写操作、暂存区操作、stash、rebase、reset、checkout、commit、branch move 或 push，避免搞乱工作区和用户未整理的任务边界。
*   **状态同步机制**：
    *   一轮实现未完成前，不要根据中间态报错反复回改同一轮逻辑。
    *   固定顺序：先完成本轮实现，再统一跑正式校验，再集中处理报错。
    *   唯一例外：会阻断继续书写的基础语法错误或编辑器级硬阻塞。
*   **指令执行耐心**：
    *   命令未明确结束前，不要把中途输出当成最终结论，也不要据此修改代码。
    *   前一个命令未明确结束前，不要发送下一个命令。
    *   编译、测试、构建、数据处理等长时命令属于正常现象，尤其是涉及编译时，更要耐心等待任务完整结束后再判断。
    *   这样做是为了避免系统过载、避免并发堆积，也避免在结果尚未完整时误判问题。
*   **串行验证顺序**：
    *   先执行静态检查、类型检查、构建检查等快速验证。
    *   再执行测试、集成验证、回归验证等较重验证。

## 10. 架构设计与代码质量
*   **设计风格优先追求优雅收口**：
    *   默认不追求“最小改动”，而追求语义最清晰、结构最优雅、代码落地与清理最干净的实现。
    *   哪怕牵扯范围较大，只要代码品味与长期结构收益上值得，就可以讨论并优先考虑更干净的收口方案，不应只因改动面大就默认退回局部修补。
    *   默认优先考虑破坏性更新，直接收口旧逻辑、旧接口与旧命名，尽量减少技术债务，而不是叠加临时桥接和局部修补。
*   **拒绝兼容层**：
    *   兼容层会污染代码质量。
    *   默认倾向于 breaking update，以追求唯一正确写法。
*   **唯一性原则**：
    *   不保留旧接口、旧桥接、旧回退、旧别名、旧术语双写法。
*   **拒绝平行造轮子**：
    *   除非 Task Spec 明确要求并通过审阅，否则不得新增与现有抽象平行的替代层。
    *   新能力默认应进入现有模块边界、现有注册体系、现有配置体系与现有验证体系，而不是在调用方临时拼一套局部逻辑。
    *   若现有抽象不够好，应把改造既有抽象纳入任务范围；禁止在旧抽象旁边堆出一个只服务当前需求的新旁路。
*   **旧逻辑清理属于正式落地的一部分**：
    *   当任务涉及替换旧设计、收口旧 contract、统一术语或消除兼容层时，旧逻辑清理应直接纳入正式执行范围，而不是作为事后可选优化。
    *   `03_execution` 应提前列出需要清理的旧逻辑、旧命名、旧桥接、旧注释与旧文案。
    *   所有执行流程完成后，应再做一轮最终检查，确认旧残留是否已经收口；`04_review` 只记录结果，不定义清理协议。
*   **清晰优雅可读**：
    *   优先选择语义直观、层次清晰、命名准确的实现。
    *   避免炫技式技巧、隐式行为与过度嵌套。
    *   代码应包含必要的中文注释，以提高审阅效率与长期可维护性。
*   **大文件拆分约束**：
    *   单个生产代码文件默认以 `500` 行以内为宜。
    *   超过 `500` 行时，应默认评估是否需要拆分。
    *   若预计落地后文件会明显变大，必须在 `03_execution` 先写拆分方案。
    *   若文件虽未超过 `500` 行，但已经承担多个职责，也应优先拆分。
*   **高内聚低耦合**：
    *   门面文件只做导出、编排与少量 glue code。
    *   核心算法、校验、切片、重建、转换、状态传播应拆入独立模块。

## 11. 异常处理与防御原则
*   **拒绝异常吞噬**：严禁滥用 `try: except Exception: pass` 或同类静默吞错写法。
*   **拒绝冗长防御**：不搞过度复杂的防御性编程。
*   **明确性胜过容错**：发现明确错误时，应直接报错，不做静默回退。
*   **库层硬约束**：在库层或核心层，只要能明确判定为错误，就直接返回明确异常；不要做兼容回退或静默兜底。

## 12. Jujutsu 与 Task 工作流
*   **AI 执行总原则**：
    *   AI 必须把 `jj` 视为默认版本管理入口；Git 只作为只读辅助或远端兼容层。
    *   AI 必须把版本管理目标理解为“让 Task、阶段、patch 的归属清晰”，不是追求大量分支或把历史整理成看似一次性完成。
    *   本工作流是 Task 驱动的，通常不需要分支功能；即使确实需要分支、bookmark 或发布线，也不应由 AI 擅自处理。
    *   AI 必须记住：`jj undo` 是恢复手段，不是擅自重写历史、跳过确认或省略报告的理由。

### 12.1 启动判断
*   **仓库已启用 `jj` 时**：
    *   AI 查看状态优先使用 `jj st`、`jj log`、`jj diff`。
    *   AI 整理任务边界优先使用 `jj change`，不使用 Git index、stash 或 Git 分支作为任务边界。
    *   AI 可以用 Git 只读命令辅助核对，例如 `git diff`、`git log`、`git show`。
*   **仓库未启用 `jj` 时**：
    *   AI 只允许使用 Git 只读命令核对状态和历史。
    *   AI 不得擅自执行 `jj git init --colocate`。
    *   AI 不得擅自执行 Git 写操作、暂存区操作、stash、rebase、reset、checkout、commit、branch move 或 push。
    *   若任务需要从 Git 迁移到 `jj`，AI 必须先只读盘点 staged、unstaged、untracked 与当前分支，再请用户确认迁移和拆分策略。

### 12.2 Change 拆分规则
*   **AI 默认使用稳定拆分粒度**：
    *   文档 change 命名为 `task <date> doc`。
    *   代码 change 命名为 `task <date> code`。
    *   后续补洞命名为 `task <date> patch-1`、`task <date> patch-2`。
    *   小任务可以合并为 `task <date> doc-code`。
    *   代码阶段确实过大时，才拆成 `task <date> code-stage-N`。
*   **AI 每新建一个 task change 必须设置 description**：
    *   新建 change 后应使用 `jj describe -m "..."` 设置简短英文描述。
    *   description 应尽量短，表达 task 日期、doc/code/patch 属性和核心范围。
    *   如果任务范围发生较大变更，AI 可以更新 description，使其继续反映当前 change 的真实内容。
*   **AI 不要把最终历史拆得过碎**：
    *   不要默认拆出 `pre-review fix`、`post-review fix`、`stage N` 等细碎 change。
    *   审阅修复若很小，应优先合入对应 `doc` 或 `code` change；若影响大或发生在正式基线之后，才新建 `patch-N`。
*   **AI 可以使用草稿 change**：
    *   草稿可命名为 `task <date> doc-draft`、`task <date> code-draft`。
    *   草稿整理完成后，AI 应用 `jj squash` 或 `jj split` 收口到稳定 change。
    *   草稿 change 不应长期作为正式任务边界。

### 12.3 AI 可自行执行的 `jj` 操作
*   **只读操作可以自行执行**：
    *   `jj st`：查看工作副本 change、冲突、bookmark 提示和整体状态。
    *   `jj log`：查看 change graph、父子关系、当前工作副本和 bookmark 位置。
    *   `jj diff`：查看当前 change 的文件改动。
    *   `jj show <rev>`：查看指定 change 的描述和 diff。
    *   `jj op log`：查看 operation log，用于定位可 undo / restore 的操作历史。
    *   `jj evolog -r <rev>`：查看某个 change 的演化历史，判断是否被 amend / squash / rebase 过。
    *   `jj bookmark list`：查看 bookmark 与远端跟踪状态。
*   **草稿整理可以执行，但必须谨慎使用写入命令**：
    *   `jj new <rev>`：从明确基线或父 change 新开任务 change。
    *   `jj describe -m "..."`：补充当前 change 的 Task 归属和阶段说明。
    *   `jj commit -m "..."`：结束当前 change 并开启下一个空 change。
    *   `jj split`：把混合变更拆成多个 change。
    *   `jj squash -i`：把当前 change 的部分内容并入父 change。
    *   `jj squash --into <rev>`：把草稿 change 或当前修补补入明确的稳定 change。
*   **执行前置要求**：
    *   AI 若无法判断某个文件或 hunk 属于哪个 Task，必须暂停询问，不能靠猜测拆分。
    *   AI 对写入命令有疑问或顾虑时，必须先和用户讨论。
    *   AI 执行任何非只读 `jj` 命令后，最终报告必须解释命令、目的和影响范围。

### 12.4 `jj rebase` 使用边界
*   **AI 不得把 `jj rebase` 当作普通整理命令**：
    *   `jj rebase` 只用于依赖关系清楚、冲突风险低、语义边界不变的 task stack 重排。
    *   rebase 前必须说明 source、destination、受影响后代、预期目的。
    *   rebase 后必须核对 `jj log` 与相关 `jj diff`。
*   **允许 AI rebase 的场景**：
    *   低影响修改补入 `task <date> doc` 或 `task <date> code` 后，让后续 change 继续接在新版本之后，且没有明显会冲突的后续任务。
    *   用户已经确认任务依赖顺序，AI 只是把 change 接到正确父节点之后。
    *   rebase 不改变 contract、测试真值、失败语义、公开接口和跨 Task 依赖判断。
    *   被 rebase 的 stack 属于用户当前可改写的个人 task stack，而不是共享主线、发布基线或他人正在依赖的 bookmark。
*   **AI 必须停下并交给用户的场景**：
    *   rebase 影响 `master/main`、release bookmark、共享 review bookmark，或他人可能已经基于旧 change 工作的 bookmark。
    *   rebase 后需要更新远端 bookmark、force-push 等价操作，或会产生 divergent changes 风险。
    *   rebase 会改变 contract、测试口径、失败语义、公开接口或任务依赖顺序。
    *   rebase 引发大面积冲突、跨多个 Task 混合冲突，或 AI 无法判断冲突归属。

### 12.5 历史补充与 patch 判断
*   **AI 可以建议补进旧 change 的条件**：
    *   旧 change 属于用户当前可改写的个人 task stack。
    *   补充内容影响很小，例如文档措辞、局部说明、明显遗漏的同轮内容。
    *   补充内容不改变已经通过人工审阅的 contract。
    *   后续 change 不存在明显机械冲突或语义冲突。
    *   即使该 change 已 push 到用户自己的 bookmark，只要用户确认这是个人 task stack，仍可按 jj clean-commit 工作流补入旧 change 后再更新 bookmark。
*   **AI 应建议新建 `patch-N` 的条件**：
    *   旧 change 已进入 `master/main`、release、已验收任务基线或共享 review 基线。
    *   旧 change 后面已有明显依赖它的后续 Task、清理任务或实现任务。
    *   补充内容来自后续新认知，而不是同一轮草稿整理。
    *   补充内容改变 contract、失败语义、测试真值、任务边界或对外接口。
    *   补充内容机械上可以 rebase，但会改变后续 Task 的审阅语义或可能引发 divergent changes。

### 12.6 用户专属或需明确授权的操作
*   **AI 禁止擅自执行以下操作**：
    *   `jj git init --colocate`
    *   `jj git fetch`
    *   `jj git push`
    *   `jj bookmark create`
    *   `jj bookmark move`
    *   `jj bookmark delete`
    *   `jj restore`
    *   `jj abandon`
    *   带 `--ignore-immutable` 的任何命令
    *   任何会更新远端 bookmark、共享 bookmark 或发布线的操作
*   **`jj undo` 的特殊规则**：
    *   AI 只能在当前轮自己刚造成问题时，说明原因后使用 `jj undo` 修正。
    *   AI 不得擅自回滚到更早 operation。
    *   非当前轮 AI 造成的问题，必须由用户确认是否 undo。
*   **发布与正式历史由用户决定**：
    *   是否初始化 `jj` 由用户决定。
    *   是否移动 `master/main` bookmark 由用户决定。
    *   是否 push 或 force-push 等价操作由用户决定。
    *   是否把某个 change 视为 Task 完成基线由用户决定。
*   **临时远端备份由用户处理**：
    *   若用户想把未收口 task stack 临时备份到远端，但不污染 `master/main` 正式历史，可使用单独的个人备份 bookmark，例如 `backup` 或 `wip`。
    *   第一次备份可由用户执行：`jj bookmark create backup -r @-`，然后执行 `jj git push --bookmark backup`。
    *   后续更新备份可由用户执行：`jj bookmark move backup --to @-`，然后执行 `jj git push --bookmark backup`。
    *   该备份线只表示临时同步点，不代表 Task 完成基线，也不代表正式发布线。
    *   AI 可以解释和提醒该备份技巧，但不得擅自创建、移动、推送或删除 `backup` / `wip` bookmark。

### 12.7 最终报告要求
*   **AI 使用 `jj` 后必须报告**：
    *   只要执行过任何非只读 `jj` 命令，最终总结必须列出命令、目的、影响到的 change 或文件范围。
    *   如果命令产生冲突、自动重基、遗留待处理状态或需要人工确认，必须明确写出。
    *   如果本轮只执行了只读 `jj` 命令，可以简要说明“未执行非只读 `jj` 命令”。


# 本项目特有的 Spec 规范
*   **本章仅适用于当前仓库**：
    *   若将本 `AGENTS.md` 复用到其他项目，可只复制前面的通用协作协议，不复制本章。
*   **项目背景**：
    *   本项目是一个基于 `PyO3` 的量化交易回测框架。
    *   `Rust` 承载核心计算逻辑与关键类型真值。
    *   `Python` 承载上层编排、策略表达、结果消费与灵活调用。
*   **当前仓库的版本管理协作口径**：
    *   当前仓库按个人项目处理，不按多人团队协作场景设计默认流程。
    *   默认不需要考虑协作者基于同一 change 并行修改造成的协作冲突。
    *   当前仓库基本不需要开分支；任务边界应优先通过 `jj change stack` 与 `doc/tasks/` 表达。
    *   若极少数场景确实需要分支、bookmark、push 或发布线调整，必须由用户明确处理或明确授权。
*   **当前仓库的最小 `jj` 工作流**：
    *   当前仓库先按最小模型理解：不用分支，不主动使用 bookmark；用一个个 `change` 管理 Task。
    *   一个 `change` 约等于一个可继续修改的 commit；一个 Task 通常对应 `task <date> doc`、`task <date> code` 或 `task <date> patch-N`。
    *   日常只需要优先使用：
        1. `jj st`：看当前状态。
        2. `jj log`：看 change stack。
        3. `jj diff`：看当前 change 改动。
        4. `jj new`：开一个新的 change。
        5. `jj describe -m "task <date> doc"`：给当前 change 写简短英文描述。
        6. `jj commit -m "task <date> doc"`：结束当前 change，并开启下一个空 change。
    *   需要把当前 change 推到远端时，优先记住 `jj git push --change @`。
    *   如果当前 `@` 是刚 `jj commit` 后的空 change，真正要推的是上一个 change，则使用 `jj git push --change @-`。
    *   bookmark 只理解为远端需要的名字；本仓库日常不要求用户先掌握 bookmark，也不要求 AI 主动引入 bookmark 工作流。
*   **当前仓库对 Spec 阶段的偏好**：
    *   复杂链路必须先充分讨论方案，再进入正式 Spec。
    *   对关键算法，Spec 应尽量给出公式、严格定义、伪代码或步骤化描述。
    *   目标是让人工在审阅阶段就能完成逻辑闭环判断，而不是只看到接口骨架。
*   **当前仓库的验收标准**：
    *   以“实现是否与最新 Task Spec 一致”为准。
    *   默认不做逐行人工审码，而是依赖人工审阅 Spec、AI 审阅实现与验证报告。
*   **当前仓库对非 Task 文档的默认处理方式**：
    *   `doc/tasks/` 之外的 `doc` 文档默认视为旧文档或历史上下文。
    *   Task 执行时默认无视这些旧文档，不主动读取、不主动对齐、不主动维护。
    *   若旧文档与最新 Task Spec 或当前源代码冲突，以最新 Task Spec 与当前源代码为准。
    *   只有在用户明确要求，或当前 Task 明确引用某份旧文档时，才需要额外读取与处理。
    *   例外是 `doc/structure/pyo3_interface_design.md`：它属于 Spec 工作流建立前留下的接口规范，目前仍作为当前仓库的接口真值之一，直到被新的 Task Spec 明确替代。
*   **当前仓库的兼容与破坏性更新策略**：
    *   本仓库在通用“单一正式口径”原则基础上，进一步偏向破坏性更新。
    *   默认倾向于破坏性更新，不保留兼容层。
    *   不保留旧接口、旧桥接、旧回退、旧别名、旧术语双写法。
    *   若确需兼容，必须在 Task Spec 与 `03_execution` 中明确说明理由、范围与退出条件；没有极强理由，不允许加兼容层。
*   **当前仓库的旧痕迹清扫协议**：
    *   本仓库在通用“旧逻辑清理应纳入执行计划”原则基础上，要求更明确的基线扫描、清理清单与最终扫描。
    *   每个 breaking task 都必须在 `03_execution` 写明 `Legacy Kill List`。
    *   在 `03_execution` 规划阶段，就应先做一轮旧痕迹基线扫描，提前识别需要清理的旧类型、旧字段、旧函数、旧兼容、旧注释措辞、旧测试口径。
    *   `Legacy Kill List` 应基于这轮基线扫描整理出来，作为后续执行阶段的明确清理目标。
    *   所有执行流程完成后，必须再做一轮旧痕迹最终扫描；默认扫描范围是主源码目录、主要脚本目录与类型/接口定义目录。
    *   旧痕迹不仅包括运行时兼容逻辑，也包括旧命名、旧注释、旧术语、旧文案残留。
    *   在最终扫描确认清理完成前，不得宣称 breaking 更新已完全落地。
    *   `04_review` 不定义清扫协议，只记录清扫结果、残留问题与验收结论。
*   **当前仓库的接口与错误体系一致性**：
    *   涉及 Rust 对外接口、Python 包装层、错误映射或异常语义时，必须先阅读并对齐 `doc/structure/pyo3_interface_design.md`。
    *   Rust 错误处理必须优先复用当前仓库既有错误体系，例如 `QuantError`、`BacktestError` 及其现有 PyO3 映射。
    *   禁止为同一类问题平行新增第二套接口风格或第二套错误系统。
    *   若确需调整接口或错误语义，应作为明确的项目级变更处理，并在 Task Spec 中写清迁移与影响范围。
*   **当前仓库的类型系统约束**：
    *   Rust 是唯一的核心类型真值来源。
    *   核心输入输出类型必须优先在 Rust 端定义。
    *   Python 类型存根通过 `pyo3-stub-gen` 自动生成。
    *   Python 侧禁止定义与 Rust 对应的镜像输入输出类型。
    *   类型变更后，应先修改 Rust，再执行 `just stub` 同步 `.pyi`，然后执行 `just check` 做静态检查。
*   **当前仓库的模板化链路约束**：
    *   回测、策略表达、指标计算、信号判断、扫描执行与数据处理必须优先复用仓库已有正式链路。
    *   策略层只负责声明参数、信号模板、组合关系与执行配置；不得在策略层自行计算 EMA、KC、ATR、MA 或其他技术指标来绕过正式指标系统。
    *   正式回测与扫描中的指标必须来自 Rust 内置指标系统，并通过仓库既有模板化回测链路被消费。
    *   信号条件必须优先通过仓库既有信号模板、表达式模板或统一执行入口表达；不得为单个策略平行维护第二套解析、计算、触发或验证系统。
    *   若需要新增指标、新信号能力或新执行能力，必须扩展现有内置系统和模板化链路，再让策略或配置消费该能力。
    *   临时研究、debug 或一次性验证代码必须限定在明确的非正式范围内；未经 Task Spec 纳入，不得进入正式策略、正式配置或正式回测入口。
*   **当前仓库的代码与验证入口约束**：
    *   务必先阅读 `justfile`，优先使用 `just` 作为统一入口。
    *   当前仓库要求串行验证：先 `just check`，再 `just test`。
    *   严禁一边修改代码一边跑正式校验；应在本轮逻辑修改完成后统一执行。
*   **当前仓库的删除约束**：
    *   禁用 `rm` / `rmdir`。
    *   删除文件应优先使用 `trash` 或其他可恢复删除方式。
*   **当前仓库的指标对齐标准**：
    *   内置指标系统是正式指标入口；新增指标必须进入 Rust 指标注册、输出列契约、warmup contract 与指标测试体系。
    *   禁止在策略、扫描、Python 编排或配置层用 `pandas`、`pandas-ta`、手写公式或临时 helper 计算指标，并把结果当成正式回测指标使用。
    *   回测引擎中的指标实现应优先以 `pandas-ta (talib=true)` 为基准。
    *   若 `pandas-ta (talib=true)` 不可用或该指标缺失，则退到 `pandas-ta (talib=false)` 作为基准。
    *   禁止直接使用原始 `talib` 作为正式对齐真值。
    *   调用方式应统一采用 `import pandas_ta as ta; ta.xxx(...)`，不要使用 `ohlc.ta.xxx(...)` 风格。
    *   当前仓库应优先使用原版 `pandas-ta` 作为正式对齐基准。
    *   只有在原版 `pandas-ta` 确实缺失某个指标时，才允许退到 `pandas-ta-classic`。
    *   原因是 `pandas-ta-classic` 相比原版存在较多未逐项验证的实现调整，部分旧指标行为与原版并不一致，因此不应默认把新版当成真值源。
    *   当前仓库的指标对齐优先级固定为：
        1. `pandas-ta (talib=true)`
        2. `pandas-ta (talib=false)`
        3. `pandas-ta-classic (talib=true)`
        4. `pandas-ta-classic (talib=false)`
*   **当前仓库的 notebook 与策略脚本分层规则**：
    *   `.py` 主要供 AI 调试与自动化执行。
    *   `ipynb` 主要供人工研究与交互分析。
    *   AI 默认不得直接读取 `ipynb`，除非用户明确要求。
    *   即使用户明确要求读取 `ipynb`，也应优先只读取 cell 代码，不读取 cell 输出结果。
    *   原因是输出结果中可能包含大表格、图表或其他高体积内容，容易无意义地撑爆上下文。
*   **当前仓库的 research 策略执行入口**：
    *   `py_entry/strategy_hub/search_spaces` 下的 research 策略，统一使用 `just workflow ...`。
    *   不要直接执行策略模块文件，也不要虚构不存在的命令。
    *   `just workflow` 只能顺序运行，不要在外层再做并发调度。
    *   原因是该入口内部已经实现了并发处理，外层再次并发容易造成资源过载与结果混乱。
*   **当前仓库的 research 结果摘要口径**：
    *   AI 应优先摘要 `just workflow` 生成的 JSON 日志，而不是只贴终端输出。
    *   每条 `walk_forward` 结果至少应列出 `strategy_name`、`symbol`、`performance.calmar_ratio_raw`、`performance.total_return`、`performance.max_drawdown`。
    *   默认按 `performance.calmar_ratio_raw > performance.total_return > -performance.max_drawdown` 比较，`performance.calmar_ratio` 仅作辅助参考。
*   **当前仓库的执行入口与验证入口应优先复用仓库现有脚手架**：
    *   若仓库已经定义了 `justfile` 且对应任务可以通过 `just` 完成，则必须优先使用 `just` 命令。
    *   不要绕开 `just` 直接调用底层工具，避免入口不一致、环境不一致或执行口径漂移。
    *   只有在 `just` 明确不存在对应入口，或当前任务确实无法通过 `just` 完成时，才允许退回到仓库其他统一脚手架或底层命令。
    *   具体命令属于本仓库实现细节，应写在任务的 `03_execution` 或项目结构文档中，而不写进通用流程主干。
