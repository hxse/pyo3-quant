# 问题背景

## 1. 当前问题

当前指标测试系统已经走对了大方向：测试通过正式回测链路执行到 `ExecutionStage.Indicator`，由 Rust 内置指标系统产生 engine 输出，再在 Python 测试层与 `pandas-ta` 结果对比。

问题在于对比测试的表达能力不足：

1. 多数指标只覆盖少量参数点。
2. 数据场景主要依赖同一套生成数据。
3. `assert_mode_pandas_ta` / `assert_mode_talib` 只能表达“应该一致”或“至少有一列不同”，语义过粗。
4. 缺失值 mask、输出列映射、oracle 选择、容差和预期不一致原因没有集中显式声明。
5. 未来新增指标时，容易复制旧模板并继续放大测试口径漂移。

## 2. 为什么先做这个任务

`2026-04-16 extensible builtin indicators` 会新增长期内置指标扩展协议，并以 KC 作为首个新增指标实例。

如果先实现 KC，再回头重构测试系统，KC 很可能先落在旧测试口径上；后续再迁移测试时，既要判断 KC 实现是否正确，又要判断测试重构是否正确，审查成本会叠加。

因此本任务应先于 04-16 代码执行完成。

## 3. 核心风险

测试框架本身不是普通业务代码。本项目不为测试框架再建立一套 meta-test；那会制造新的测试维护负担，也无法证明整体 oracle 选择、对比口径和迁移语义没有漂移。

本任务必须用三层方式控制风险：

1. Spec 先冻结测试系统的正式语义。
2. 语义新增与冻结清单作为最低审查基线。
3. post-review 必须结合 `git diff` 做全面审查，把审查做好，不能只按清单机械打勾。
