# TODO: 结构搜索器设计方案 (Structure Searcher)

本文档描述一个**受约束的策略结构搜索器**。它不是“全能自动发明策略”的黑箱，而是在项目现有的**信号模板 + 参数传递**体系上，自动组合有限的策略部件，用于大规模结构探索。

它的定位是：

*   **未来可选的研究能力增强项**
*   **不是当前框架可运行性的刚需能力**

也就是说，即使暂时不实现这个结构搜索器，项目现有 workflow 也已经足够支撑正常的策略研究与回测流程；差别主要只是：

*   需要人工手写更多策略
*   结构空间的探索效率较低

---

## 1. 目标与边界

### 1.1 目标

结构搜索器的目标不是自动优化参数，而是：

*   自动组合不同的指标角色与风控开关。
*   在**默认参数**下快速探索大量策略结构。
*   尽量排除“纯参数调优”导致的过拟合。
*   为后续更贵的研究步骤（如多品种复筛、向前测试）提供候选策略。

### 1.2 为什么第一阶段只做结构搜索

第一阶段固定默认参数，有几个明确好处：

*   搜索速度更快，适合大规模探索。
*   结果更容易解释，便于人工审阅。
*   更容易判断“结构本身有没有 Alpha”，而不是参数是否刚好拟合了样本。
*   可以把向前测试留给少量候选做深度研究，而不是拿来做海选。

### 1.2.1 搜索器的总原则：依托高性能做大空间暴力搜索

本项目的回测引擎本身已经足够快，因此结构搜索器的总原则不是“靠大量前置经验约束缩小搜索空间”，而是：

*   **只保留最小必要约束**
*   **依托高性能回测底座**
*   **直接用遗传算法对较大的结构空间做暴力搜索**

也就是说：

*   不应过早把大量主观经验写进搜索空间约束。
*   不应靠人工预判“哪些组合大概率没用”来代替机器搜索。
*   更合理的方式是让搜索器先尽量探索，再用后置筛选把差的结构淘汰掉。

但这不等于完全放弃结构化设计。即使搜索空间放得较大，**指标和组件的分层仍然必须保留**，否则搜索器会退化成无边界的杂乱组合器，最终既不利于搜索，也不利于编译和审阅。

### 1.3 不做什么

第一阶段明确**不做**：

*   任意公式树自动生成。
*   自由文本策略生成。
*   大量连续参数优化。
*   任意深度的布尔嵌套逻辑。

搜索器只在白名单组件和固定骨架内工作。

## 2. 分层原则：Rust 作为唯一解释器，外层语言只提供配置

### 2.1 Python / TS 层负责什么

Python / TS 等外层语言只负责提供**搜索空间配置文件**，不负责解释搜索空间。

这里的“配置文件”必须理解成：

*   **结构化对象 / 字典配置**
*   **可以序列化为 JSON / YAML / TOML**
*   **不采用字符串模板或自定义 DSL**

原因是：

*   字符串模板不利于强校验。
*   字符串模板不利于跨语言传递。
*   搜索空间本身包含层次、槽位、候选值、约束、GA 参数和评估参数，天然更适合结构化对象表达。
*   Rust 解析结构化配置更稳，也更容易保持唯一真值。

外层负责：

*   组织搜索空间配置。
*   调用 Rust 搜索接口。
*   消费搜索结果。
*   展示、落盘和上层编排。

也就是说，外层语言只负责：

*   **传配置**
*   **拿结果**

而不负责：

*   遗传算法迭代。
*   genome 解码。
*   候选结构解析。
*   候选编译。
*   批量评估。

### 2.2 Rust 层负责什么

Rust 层作为**唯一解释器**，负责完整的搜索闭环。

Rust 层负责：

*   解析搜索空间配置。
*   遗传算法迭代。
*   genome 编码与解码。
*   搜索空间合法性校验。
*   把 genome 解析成候选结构。
*   再把候选结构编译成：
    *   `IndicatorsParams`
    *   `SignalParams`
    *   `BacktestParams`
    *   `SignalTemplate`
*   自动批量评估。

同时 Rust 层继续作为执行真值层，负责：

*   `DataPack / ResultPack` 真值。
*   指标、信号、回测、绩效执行。
*   `ranges / mapping / warmup / walk_forward / stitched`。
*   给定参数和模板后的稳定回测结果。

### 2.3 两层如何衔接

两层之间的关系应该是：

*   Python / TS 交给 Rust 一份“搜索空间配置文件”。
*   Rust 在这份搜索空间上完成：
    *   配置解析
    *   遗传算法迭代
    *   genome 解析
    *   候选结构编译
    *   批量回测评估

也就是说：

*   外层语言提供配置。
*   Rust 解释配置并执行搜索。

### 2.3.1 搜索空间配置示意

下面只是一个最小示意，重点是说明配置应为**结构化对象**，而不是字符串模板：

```python
search_space = {
    "layers": {
        "context_layer": {
            "trend_slots": {
                "min": 0,
                "max": 2,
                "candidates": ["ema_trend", "supertrend", "hma_trend"],
            },
            "filter_slots": {
                "min": 0,
                "max": 1,
                "candidates": ["atr_gate", "adx_gate"],
            },
            "htf_slots": {
                "min": 0,
                "max": 1,
                "candidates": ["htf_ema_confirm"],
            },
        },
        "signal_layer": {
            "trigger_slots": {
                "min": 1,
                "max": 2,
                "candidates": ["rsi_reversion", "bbands_touch", "macd_cross"],
            },
        },
        "exit_layer": {
            "exit_slots": {
                "min": 1,
                "max": 1,
                "candidates": ["rsi_mid_exit", "signal_flip"],
            },
        },
        "risk_layer": {
            "risk_mode": {
                "min": 1,
                "max": 1,
                "candidates": ["none", "tsl_only", "sl_tp"],
            },
            "risk_refs": {
                "min": 0,
                "max": 1,
                "candidates": ["atr", "psar"],
            },
        },
    },
    "constraints": {
        "max_total_components": 6,
        "max_htf_components": 1,
    },
    "ga": {
        "population_size": 200,
        "generations": 50,
        "mutation_rate": 0.1,

    },
    "evaluation": {
        "sort_metric": "calmar_ratio_raw",
        "min_total_trades": 20,
    },
}
```

这个配置由 Python / TS 组织并传给 Rust；Rust 负责：

*   解析配置
*   建立 genome 编码
*   遗传算法迭代
*   候选结构编译
*   批量评估

### 2.4 为什么必须让 Rust 负责解析空间

让 Rust 负责解析搜索空间，有两个关键好处：

1. **避免漂移**
*   搜索空间解释逻辑只存在一份。
*   不会出现 Python 一套、TS 一套、后续再各自分叉的情况。

2. **天然支持跨语言**
*   以后不只是 Python 可以调。
*   TypeScript 也可以直接传同一份配置给 Rust。
*   不需要再为 TS 重写一套搜索空间解析器。

因此，这里的原则应该写死：

*   **搜索空间配置可以跨语言传入**
*   **搜索空间解释逻辑只能存在于 Rust**

## 3. 搜索器的核心思想：搜结构，不搜代码

### 3.1 搜索器不是自动写策略代码

搜索器不生成 Python 策略文件，也不拼接源码。它只生成一个中间对象：

```python
candidate = {
    "context": {
        "trends": ["ema_trend"],
        "filters": ["atr_gate"],
        "htf_filters": [],
    },
    "signal": {
        "triggers": ["rsi_reversion"],
    },
    "exit": {
        "exits": ["rsi_mid_exit"],
    },
    "risk": {
        "risk_mode": "tsl_only",
        "risk_refs": ["atr"],
    },
    "side_mode": "both",
}
```

这个对象只表示：

*   选了哪些组件。
*   这些组件各自承担什么角色。

它还不是可执行策略。

### 3.2 搜索器真正要做的事

搜索器负责：

1. 生成一个候选结构 `candidate`。
2. 校验这个结构是否合法。
3. 把它编译成现有回测系统可接受的配置。
4. 跑回测。
5. 排序、筛选、记录。

## 4. 搜索空间应该如何设计

### 4.1 不是“每类只能选一个”，也不是“任意逻辑树”

最优雅的设计不是：

*   `trend` 只能选一个；
*   `trigger` 只能选一个；

也不是：

*   允许任意层级任意嵌套。

更合理的方式是：

*   **固定层次**
*   **每层允许少量槽位**
*   **每层有职责边界**
*   **每层内部允许有限组合**

### 4.2 统一骨架

建议固定成 4 层：

```text
context_layer
signal_layer
exit_layer
risk_layer
```

#### `context_layer`

负责环境和方向限制：

*   `trend_slots`: 0~2 个
*   `filter_slots`: 0~1 个
*   `htf_slots`: 0~1 个

示例组件：

*   `ema_trend`
*   `supertrend`
*   `adx_gate`
*   `atr_gate`
*   `htf_ema_confirm`

#### `signal_layer`

负责真正的入场触发：

*   `trigger_slots`: 1~2 个

示例组件：

*   `rsi_reversion`
*   `bbands_touch`
*   `macd_cross`
*   `stoch_turn`
*   `price_breakout`

#### `exit_layer`

负责主动离场逻辑：

*   `exit_slots`: 1~1 个（第一版先固定 1 个）

示例组件：

*   `signal_flip`
*   `rsi_mid_exit`
*   `ma_cross_back`
*   `bbands_middle_touch`

#### `risk_layer`

负责被动风控：

*   `risk_mode`: 必选 1 个
*   `risk_refs`: 0~1 个

示例：

*   `none`
*   `sl_only`
*   `tp_only`
*   `tsl_only`
*   `sl_tp`
*   `sl_tp_tsl`

`risk_ref` 示例：

*   `atr`
*   `psar`
*   `swing`
*   `fixed_pct`

### 4.3 第一版的推荐搜索空间

第一版不必过宽，也不要过死。建议：

*   `trend_slots`: 0~2
*   `filter_slots`: 0~1
*   `htf_slots`: 0~1
*   `trigger_slots`: 1~2
*   `exit_slots`: 1
*   `risk_mode`: 1
*   `risk_refs`: 0~1

这样已经能支持：

*   `trend + trend`
*   `trend + filter`
*   `trigger + trigger`
*   `trigger + filter`

同时不会爆炸成任意逻辑树。

## 5. 如何做“不过分紧”的约束

### 5.1 原则

约束只做必要的，不做过度经验裁剪。

原因是：

*   回测引擎本身高性能。
*   第一版搜索器直接用遗传算法。
*   搜索空间不需要为了性能而被人为压得过小。

因此，这里的约束目标不是“尽量提前缩小搜索空间”，而是：

*   保证结构可编译。
*   保证结构可执行。
*   保证结构没有明显自相矛盾。

除此之外，不应加入过多主观预筛。

### 5.2 `family + role` 只作为辅助元数据

`family + role` 可以保留，但不应该成为第一版搜索器的强硬约束中心。

它更适合做：

*   组件分类。
*   日志展示。
*   搜索结果分析。
*   后续想进一步收紧搜索空间时的辅助信息。

而不是第一版就靠它强行卡死组合。

如果某些组件暂时没有完整的 `family` 定义，也不应阻碍进入搜索空间。

### 5.3 合法性规则

建议第一版只保留最小必要硬规则：

1. `signal_layer` 必须至少有 1 个 `trigger`。
2. 若 `risk_mode` 依赖某类 `risk_ref`，则必须显式存在对应引用。
3. `exit_layer` 必须至少有 1 个 `exit`。
4. 高周期组件总数最多 1 个。
5. 总组件数不超过预设预算，例如 5~6 个。

### 5.4 允许但不放任

因此，下面这些组合应该允许：

*   `ema_trend + supertrend`
*   `rsi_reversion + bbands_touch`
*   `ema_trend + atr_gate + rsi_reversion`

但下面这些应直接剔除：

*   没有 trigger 只有 filter 的结构。
*   开启 `tsl_atr` 却没有 `atr` 引用。
*   总组件数远超预算的怪物结构。

如果后续发现某些高相关、明显无效的组合确实太多，再逐步补充白名单/黑名单规则即可，不必在第一版提前过度设计。

### 5.5 放开搜索空间，不等于取消分层

虽然总原则是放开搜索空间、减少经验裁剪，但**指标分层仍然必须保留**。

也就是说：

*   可以减少“哪些组件不能一起出现”的主观限制。
*   但不能取消：
    *   `context_layer`
    *   `signal_layer`
    *   `exit_layer`
    *   `risk_layer`

因为分层本身不是为了缩小搜索空间，而是为了：

*   保证结构清晰。
*   保证编译过程稳定。
*   保证结果可审阅。
*   保证搜索器不会退化成无边界的杂乱逻辑拼接。

## 6. 自动组合与遗传算法

### 6.1 第一阶段优先上遗传算法

自动组合和遗传算法不是一回事，但在当前这个项目里，第一阶段更适合直接引入遗传算法。

原因很直接：

*   搜索空间很大。
*   单纯随机采样的命中率可能太低。
*   小规模枚举只能覆盖很窄的一部分空间。
*   已经有足够快的回测底座，可以支撑遗传搜索的高频评估。

因此，第一阶段更合理的做法是：

*   直接以遗传算法作为第一版主搜索器。
*   不再把随机采样或小规模枚举作为正式搜索方案。

### 6.2 候选生成器与候选评估器分离

建议把搜索器拆成两层：

#### 候选生成器

负责生成 genome / candidate：

*   遗传算法

#### 候选评估器

负责：

*   解析 genome
*   编译 `candidate`
*   调回测引擎
*   评分
*   排名
*   落盘

这样搜索器的主生成器虽然是遗传算法，但后面的执行链路依然不和遗传算法具体实现绑定。

未来如果要：

*   调整遗传算法编码方式
*   更换选择/变异/交叉策略
*   引入更复杂的候选生成策略

都只需要替换候选生成器，不需要动后面的执行链路。

### 6.3 遗传算法何时开始介入

虽然第一阶段优先用遗传算法，但仍然应该在以下前提下开始：

*   搜索空间定义稳定
*   组件白名单稳定
*   编译器稳定
*   评分函数稳定
*   搜索日志与去重机制稳定

也就是说，不是等所有功能都做完再加遗传算法，而是先把最小闭环搭好，再让遗传算法成为主搜索器。

## 7. 如何和现有信号模板结合

### 7.1 搜索器不绕过信号模板

最关键的一点：

*   搜索器**不直接生成策略代码**
*   搜索器**不绕过现有 `SignalTemplate`**

它最终仍然必须编译成现有框架支持的：

*   `indicators_params`
*   `signal_params`
*   `backtest_params`
*   `signal_template`

### 7.2 Rust 层负责把组件编译成模板片段

Rust 层在解析搜索空间后，需要把每个组件视为“模板片段生成器”，至少得到下面几样东西：

1. 它需要哪些指标。
2. 它需要哪些信号参数。
3. 它往 `SignalTemplate` 的哪个槽位追加 comparisons。
4. 它是否需要补充或修改 `BacktestParams`。

例如：

*   `rsi_reversion`
    *   需要一个 RSI 指标。
    *   需要 `rsi_oversold` / `rsi_mid` 两个信号参数。
    *   往 `entry_long / entry_short` 追加 RSI 比较表达式。

*   `ema_trend`
    *   需要一个 EMA 指标。
    *   不一定需要额外信号参数。
    *   往 `entry_long / entry_short` 追加方向过滤表达式。

*   `tsl_only`
    *   不一定改 `SignalTemplate`。
    *   但要补充 `BacktestParams`。

### 7.3 组件最终都要汇总成一个 `SignalTemplate`

示意流程：

1. Rust 层先把 genome 解析成 `candidate`。
2. Rust 层编译器遍历所有组件。
3. 聚合：
   * `indicators_params`
   * `signal_params`
   * `backtest_params`
   * `SignalTemplate` 各槽位 comparisons
4. 输出一个完整的 `VariantPayload`

最终仍然复用现有 `Backtest` 入口。

## 8. 参数如何传递

### 8.1 第一阶段参数全部用默认值

第一阶段不做参数优化，所以组件编译时直接给默认参数：

*   指标参数：默认值
*   信号阈值：默认值
*   风控参数：默认值

而不是给一堆 `optimize=True` 的范围。

### 8.2 参数来源分三类

#### 指标内部参数

例如：

*   `rsi.period = 14`
*   `ema.period = 200`
*   `atr.period = 14`

#### 信号阈值参数

例如：

*   `rsi_oversold = 30`
*   `rsi_mid = 50`
*   `atr_gate_min = 某默认值`

#### 风控参数

例如：

*   `tsl_atr = 2.0`
*   `sl_pct = 0.02`
*   `tp_pct = 0.04`

### 8.3 参数命名必须避免冲突

组件编译时必须保证：

*   指标 key 不冲突
*   `signal_params` 名称不冲突
*   风控字段有明确唯一来源

例如不能两个组件都使用模糊的 `threshold`，必须写成：

*   `rsi_oversold`
*   `atr_gate_min`
*   `bbands_width_min`

## 9. 搜索空间与 Rust 遗传算法的关系

### 9.1 外层语言只提供配置，不解析空间

外层语言不负责：

*   genome 解码
*   候选结构展开
*   候选编译

这些都交给 Rust 层。

### 9.2 Rust 负责解析搜索空间并完成遗传算法

Rust 层负责：

*   根据外层给出的搜索空间配置，建立 genome 编码。
*   在 genome 空间上做遗传算法迭代。
*   把 genome 解码成候选结构。
*   把候选结构编译成现有回测引擎可执行的参数和模板。
*   自动做批量评估。

### 9.3 为什么这样分层

这样分层的好处是：

*   外层语言足够轻，只负责传配置和拿结果。
*   Rust 层把“搜索 + 解析 + 编译 + 评估”收成一个闭环。
*   不会出现 Python、TS 和 Rust 各维护一套候选解释逻辑。

## 10. 一个完整的编译示意

### 10.1 输入候选

```python
candidate = {
    "context": {
        "trends": ["ema_trend"],
        "filters": ["atr_gate"],
        "htf_filters": [],
    },
    "signal": {
        "triggers": ["rsi_reversion"],
    },
    "exit": {
        "exits": ["rsi_mid_exit"],
    },
    "risk": {
        "risk_mode": "tsl_only",
        "risk_refs": ["atr"],
    },
    "side_mode": "both",
}
```

### 10.2 编译后目标

编译器输出：

*   `IndicatorsParams`
*   `SignalParams`
*   `BacktestParams`
*   `SignalTemplate`

例如最终逻辑可表现为：

*   `RSI` 反转作为入场主触发
*   `EMA` 提供趋势方向过滤
*   `ATR` 提供波动过滤
*   `RSI` 回中线作为主动离场
*   `TSL` 作为被动风控

### 10.3 重要边界

搜索器只是：

*   选组件
*   编译组件

真正的：

*   数据构建
*   warmup
*   信号生成
*   回测
*   绩效

仍然全部走现有引擎。

## 11. 评价流程建议

### 11.1 第一层：快速海选

第一层的目标就是放开搜索范围，做大规模结构海选。

特点是：

*   不限品种。
*   不限策略结构。
*   直接用默认参数。
*   直接用遗传算法大规模探索。

这一层的重点不是做严格验证，而是：

*   尽快发现值得继续研究的候选结构。
*   尽量扩大搜索覆盖面。

### 11.2 第二层：少量候选再进 WF

第二层再把第一层表现好的少量候选送入 `walk_forward` 做更严格研究。

但这一层**不是本次任务范围**。

本次任务只讨论：

*   第一层结构搜索器如何设计。
*   第一层候选如何编译与评估。

### 11.3 候选评估口径

### 11.3.1 排序口径

候选评估器的主排序指标只保留一个：

*   `calmar_ratio_raw`

如果只能用一个值做排序，就直接按 `calmar_ratio_raw` 排序。

### 11.3.2 过滤口径

交易次数不参与主排序，只作为过滤条件使用。

也就是说：

*   先对候选结构跑回测。
*   若 `total_trades` 低于最小门槛，则直接淘汰。
*   通过交易次数门槛后，再按 `calmar_ratio_raw` 排序。

### 11.3.3 为什么这样设计

这样做的好处是：

*   主排序口径单一，结果更清楚。
*   不把多个指标强行揉成一个复杂评分函数。
*   交易次数承担“去掉样本太少的伪好结果”这一层职责。
*   卡尔马比率继续作为结构海选的主目标。

## 12. 推荐的第一版落地范围

第一版建议克制：

### 12.1 先做这些

*   搜索空间配置结构定义
*   Rust 层 genome 编码与解码
*   合法性校验
*   Rust 遗传算法
*   编译为现有 `VariantPayload`
*   Rust 侧自动批量评估
*   记录结果并排序

### 12.2 暂时不要做这些

*   自由逻辑树
*   自动发明新指标
*   参数联合优化
*   复杂 regime 切换

### 12.3 遗传算法实现策略

第一版建议 **Rust 自己实现一个小型、专用的遗传算法内核**，而不是先依赖第三方 GA 库。

原因是：

*   本项目的搜索对象不是标准连续数值向量，而是带有槽位、约束和编译逻辑的结构化 genome。
*   真正复杂的部分不是 GA 数学本身，而是：
    *   搜索空间解析
    *   genome 编解码
    *   合法性校验
    *   候选结构编译
    *   批量回测闭环
*   通用第三方 GA 库往往更适合标准优化问题，不一定适合本项目这种高度定制的结构搜索。
*   提前引入第三方库，反而容易带来额外抽象、接口不贴合、日志结构受限、非法个体修复不顺手等问题。

因此，第一版的建议是：

*   只实现一个最小可用的专用 GA 内核。
*   先支持：
    *   初始种群生成
    *   fitness 批量评估
    *   elite 保留
    *   选择
    *   交叉
    *   变异
    *   去重
    *   合法性校验
*   不提前追求复杂 GA 变种。

后续只有在下面两个条件同时成立时，才再考虑是否引入第三方库：

1. genome / fitness / 日志接口已经非常稳定。
2. 第三方库能明显减少代码量，并且不会破坏当前搜索器的唯一真值和批量评估闭环。

## 13. 总结

结构搜索器最重要的定位是：

*   它不是一个自由写策略的 AI。
*   它不是一个自动过拟合机器。
*   它是在现有信号模板与参数传递体系上，做**受约束的结构探索**。

最核心的设计原则只有三句：

1. 外层语言只提供配置，Rust 作为唯一解释器完成遗传搜索闭环。
2. 搜索结构，不搜索代码。
3. 先固定默认参数做海选，再把向前测试留给少量候选。
